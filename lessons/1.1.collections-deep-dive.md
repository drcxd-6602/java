# Session 1.1 — Collections Deep Dive

**Goal:** Know the Java collections you'll use every day — List, Set, Map, Queue — when to use which, and the `equals`/`hashCode` contract (why `HashMap` breaks without it). You know C# collections, so I map each one to its C# twin.

---

## 1. The big picture

A **collection** holds a group of objects. Java's collections live in `java.util`. They sit under a few main **interfaces** (the "what it does") with concrete **classes** (the "how it's built").

```
Iterable
  └── Collection
        ├── List   (ordered, allows duplicates)        -> ArrayList, LinkedList
        ├── Set    (no duplicates)                      -> HashSet, LinkedHashSet, TreeSet
        └── Queue  (process in an order)                -> ArrayDeque, PriorityQueue

Map   (key -> value, NOT a Collection)                  -> HashMap, LinkedHashMap, TreeMap
```

Note: **`Map` is not a `Collection`.** It's a separate family. It stores pairs, not single items.

**Golden rule (like C#):** declare the variable as the **interface**, create the **class**.

```java
List<String> names = new ArrayList<>();   // program to the interface
```

C# does the same: `IList<string> x = new List<string>();`.

---

## 2. List — ordered, allows duplicates

A `List` keeps insertion order and lets you access by index. Two main classes:

| Class        | Backed by            | Fast at                  | Slow at                     | C# twin         |
| ------------ | -------------------- | ------------------------ | --------------------------- | --------------- |
| `ArrayList`  | a resizable array    | get by index, add at end | insert/remove in the middle | `List<T>`       |
| `LinkedList` | a doubly-linked list | insert/remove at ends    | get by index                | `LinkedList<T>` |

**Use `ArrayList` almost always.** It's the default. Reach for `LinkedList` only when you add/remove at the front a lot (like a queue).

```java
List<String> products = new ArrayList<>();
products.add("ebook");
products.add("preset");
products.add("ebook");          // duplicate allowed
String first = products.get(0); // "ebook"  (index access)
```

Big-O quick view (`ArrayList`): `get(i)` = O(1), `add` at end = O(1) amortized, `add`/`remove` in middle = O(n).

---

## 3. Set — no duplicates

A `Set` cannot hold the same element twice. Adding a duplicate is silently ignored.

| Class           | Order                | C# twin                              |
| --------------- | -------------------- | ------------------------------------ |
| `HashSet`       | no order (fast)      | `HashSet<T>`                         |
| `LinkedHashSet` | insertion order kept | (LinkedHashSet has no exact C# twin) |
| `TreeSet`       | sorted order         | `SortedSet<T>`                       |

```java
Set<String> tags = new HashSet<>();
tags.add("java");
tags.add("java");   // ignored — still one "java"
System.out.println(tags.size());  // 1
```

**Use `HashSet`** when you just need uniqueness and fast lookup. Use `TreeSet` when you need them **sorted**. Use `LinkedHashSet` when you need **uniqueness + keep the order added**.

Lookup (`contains`) in `HashSet` is O(1) on average. In `TreeSet` it's O(log n) (it's a tree).

---

## 4. Map — key to value

A `Map` stores **key → value** pairs. Keys are unique. This is C#'s `Dictionary`.

| Class           | Order           | C# twin                 |
| --------------- | --------------- | ----------------------- |
| `HashMap`       | no order (fast) | `Dictionary<K,V>`       |
| `LinkedHashMap` | insertion order | (ordered dictionary)    |
| `TreeMap`       | sorted by key   | `SortedDictionary<K,V>` |

```java
Map<String, Integer> priceByProduct = new HashMap<>();
priceByProduct.put("ebook", 500);
priceByProduct.put("preset", 300);

Integer p = priceByProduct.get("ebook");        // 500
Integer missing = priceByProduct.get("course"); // null (not found)

priceByProduct.getOrDefault("course", 0);        // 0 (safe default)
priceByProduct.containsKey("ebook");             // true
```

Useful methods: `put`, `get`, `getOrDefault`, `containsKey`, `keySet()`, `values()`, `entrySet()`.

Loop a map:

```java
for (Map.Entry<String, Integer> e : priceByProduct.entrySet()) {
    System.out.println(e.getKey() + " = " + e.getValue());
}
```

**Use `HashMap`** by default. `TreeMap` when you want keys sorted. `LinkedHashMap` to keep insertion order.

---

## 5. Queue & Deque — process in an order

A `Queue` processes items in some order. A `Deque` ("deck") is a double-ended queue — add/remove at both ends.

| Class           | Behavior                                               | C# twin                             |
| --------------- | ------------------------------------------------------ | ----------------------------------- |
| `ArrayDeque`    | fast FIFO queue or LIFO stack                          | `Queue<T>` / `Stack<T>`             |
| `PriorityQueue` | always gives the "smallest" next (by order/comparator) | `PriorityQueue<TElement,TPriority>` |

```java
Queue<String> jobs = new ArrayDeque<>();
jobs.offer("email1");   // add to back
jobs.offer("email2");
String next = jobs.poll();  // "email1" (take from front, FIFO)
```

You won't use these much yet, but you'll meet `ArrayDeque` for job/task ordering later. **Do NOT use the old `Stack` class** — it's legacy and slow; use `ArrayDeque` as a stack.

## 6. Ordering: Comparable vs Comparator

To sort objects, Java needs to know "which comes first". Two ways:

### `Comparable` — the object's natural order (one, built-in)

The class implements `Comparable<T>` and defines `compareTo`. This is its *default* sort.

```java
public class Product implements Comparable<Product> {
    String name;
    int price;
    public int compareTo(Product other) {
        return Integer.compare(this.price, other.price); // sort by price
    }
}
```

`compareTo` returns: negative if this < other, 0 if equal, positive if this > other.

### `Comparator` — a separate, swappable order (many)

A `Comparator` is an outside object that defines an order. Use it to sort by different fields without changing the class.

```java
List<Product> products = ...;
products.sort(Comparator.comparingInt(p -> p.price));                 // by price
products.sort(Comparator.comparing((Product p) -> p.name));           // by name
products.sort(Comparator.comparingInt((Product p) -> p.price)
                        .thenComparing(p -> p.name));                 // price, then name
```

**C# analogy:** `Comparable.compareTo` ≈ implementing `IComparable<T>.CompareTo`. `Comparator` ≈ passing an `IComparer<T>` or a lambda to `OrderBy`.

Rule of thumb: **one natural order** → `Comparable`. **Many/other orders** → `Comparator`.

## 7. equals() and hashCode() — the most important contract

This is the heart of the session. Get it wrong and `HashSet` / `HashMap` break in ways that are very hard to debug.

### What they are

- **`equals(Object o)`** — are these two objects "the same value"? (Like C# `Equals`.)
- **`hashCode()`** — an int "bucket number" for the object. (Like C# `GetHashCode`.)

By default, `equals` compares **references** (are they the exact same object in memory?), and `hashCode` is based on the memory address. So two different objects with the same data are NOT equal by default.

### Why HashMap/HashSet need them

A `HashMap` finds a key in two steps:

1. Use `hashCode()` to jump to the right **bucket** (fast).
2. Inside that bucket, use `equals()` to find the exact key.

So if you use your own class as a **map key** or a **set element**, you MUST override both. If you don't, two objects with identical data won't be found — because they land in different buckets (different default hashCode).

### The contract (rules you must follow)

1. If `a.equals(b)` is true, then `a.hashCode() == b.hashCode()` **must** be true.
2. Equal objects must stay equal (consistent).
3. If two objects are NOT equal, their hashCodes *may* still collide — that's fine, just slower.

The killer rule is #1: **equal objects must have equal hashCodes.** If you override `equals` but not `hashCode`, you break it.

### The easy, correct way

Let the IDE or a `record` do it.

```java
// records auto-generate correct equals + hashCode from the fields
public record ProductId(String value) { }
```

Or in a normal class, use the built-in helpers:

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Product other)) return false;
    return price == other.price && Objects.equals(name, other.name);
}
@Override
public int hashCode() {
    return Objects.hash(name, price);
}
```

**Note both use the SAME fields** (`name`, `price`). That's the key: whatever fields decide equality must be the same fields that build the hashCode.

### The mutation trap (a classic bug)

If you put an object in a `HashSet`, then **change a field used in its hashCode**, the object is now in the "wrong bucket". The set can no longer find it — `contains` returns false even though it's in there.

**Rule:** don't mutate fields used in `hashCode` while the object is inside a hash-based collection. This is why keys are often **immutable** (like `String`, or a `record`).

## 8. Key terms, plain English

| Term             | Meaning                                                           |
| ---------------- | ----------------------------------------------------------------- |
| **Collection**   | A group of objects (List/Set/Queue).                              |
| **List**         | Ordered, index access, duplicates allowed. `ArrayList` = default. |
| **Set**          | No duplicates. `HashSet` = default.                               |
| **Map**          | Key→value pairs. `HashMap` = default. Not a Collection.           |
| **Deque**        | Double-ended queue. `ArrayDeque` for stack/queue.                 |
| **Comparable**   | The object's one natural sort order (`compareTo`).                |
| **Comparator**   | A separate, swappable sort order.                                 |
| **equals**       | Value equality check.                                             |
| **hashCode**     | Bucket number for hash collections.                               |
| **The contract** | Equal objects MUST have equal hashCodes.                          |

## 11. One-line recap

Use `ArrayList` / `HashSet` / `HashMap` by default; pick sorted (`Tree*`) or ordered (`Linked*`) variants when you need them. If your own class is a set element or map key, override `equals` and `hashCode` together from the same fields — and keep those fields unchanging while the object lives in the collection.
