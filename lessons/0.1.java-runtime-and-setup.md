# Session 0.1 — The Java Runtime Model & Setup

**Goal:** Understand what JDK, JRE, and JVM are. Understand how Java code goes from text to running. Get a working setup. You know C#, so I use C# to explain.

---

## 1. The big picture: how Java runs

Java does NOT turn your code straight into machine code (the 1s and 0s a CPU runs). It does it in two steps:

1. You write code in a `.java` file (plain text).
2. A compiler (`javac`) turns it into **bytecode** — a `.class` file.
3. The **JVM** reads that bytecode and runs it.

So the flow is:

```
Hello.java  --(javac)-->  Hello.class  --(java / JVM)-->  program runs
 (your code)    compile     (bytecode)       run
```

**C# is the same idea.** Your C# turns into **IL** (Intermediate Language), and the **CLR** runs it. Java's bytecode = C#'s IL. Java's JVM = C#'s CLR. Same design, different names.

### Why two steps? Why not straight to machine code?

Because machine code is different for each OS and CPU. If Java compiled straight to machine code, you'd need a separate build for Windows, Mac, Linux.

Instead, `javac` makes bytecode, which is the same everywhere. Then each OS has its own JVM that knows how to run bytecode on that OS. So:

**Write once, run anywhere.** The `.class` file works on any machine that has a JVM. This is Java's famous promise.

---

## 2. JDK vs JRE vs JVM

These three names confuse everyone. Here is the simple version. Think of boxes inside boxes.

```
+-------------------------------------------------+
|  JDK  (for developers - build + run)            |
|                                                 |
|   javac (compiler), jar, javadoc, debugger...   |
|                                                 |
|   +-----------------------------------------+   |
|   |  JRE  (for running only)                |   |
|   |                                         |   |
|   |   libraries (String, List, etc.)        |   |
|   |                                         |   |
|   |   +---------------------------------+   |   |
|   |   |  JVM  (the engine that runs     |   |   |
|   |   |       bytecode)                 |   |   |
|   |   +---------------------------------+   |   |
|   +-----------------------------------------+   |
+-------------------------------------------------+
```

- **JVM** (Java Virtual Machine): the engine. It takes bytecode and runs it. It's "virtual" because it acts like a pretend computer that understands bytecode.
- **JRE** (Java Runtime Environment): the JVM **plus** the standard libraries (the built-in classes like `String`, `ArrayList`, `Math`). This is enough to *run* a Java program, but not to *build* one.
- **JDK** (Java Development Kit): the JRE **plus** developer tools — most importantly `javac` (the compiler). This is what YOU need as a developer.

**Simple rule:** As a developer, install the **JDK**. It contains everything.

**C# analogy:**
- JVM ≈ CLR (the engine)
- JRE ≈ .NET Runtime (engine + libraries, run only)
- JDK ≈ .NET SDK (adds the compiler and tools so you can build)

---

## 3. What the JVM actually does (high level)

The JVM is not just a "player" for bytecode. It does real work while your program runs. Three parts matter now:

### a) Class loading
When your program needs a class (say `String`), the JVM finds the `.class` file and loads it into memory. It does this **lazily** — a class is loaded the first time it's actually used, not all upfront.

### b) JIT (Just-In-Time compilation)
At first, the JVM *interprets* bytecode — reads it line by line. That's a bit slow. But the JVM watches which code runs a lot ("hot" code). For that hot code, it compiles the bytecode into real machine code on the fly. So Java starts a little slow, then speeds up as it runs.

**This is why a Java server often gets faster after it's been running for a while** — the JIT has warmed up.

C# does the same: the CLR also JIT-compiles IL to machine code.

### c) GC (Garbage Collection)
You don't free memory by hand in Java (no `free()`, no `delete`). When objects are no longer used, the **Garbage Collector** finds them and frees that memory automatically.

C# has GC too, so this will feel familiar. Same idea.

You don't need GC details now. Just know: **memory cleanup is automatic.**

## 4. Key terms, plain English

| Term | Meaning |
|------|---------|
| **Bytecode** | The middle format `javac` produces. Not human code, not machine code. The JVM runs it. Like C#'s IL. |
| **`javac`** | The Java compiler. Turns `.java` into `.class`. |
| **`java`** | The command that starts the JVM and runs a `.class`. |
| **Platform independent** | Same `.class` runs on any OS with a JVM. |
| **LTS** | Long-Term Support. A Java version that gets updates for years. Use LTS for real work. Java 21 is LTS. |

---

## 5. Java versions — which one?

Java releases often, but only some are **LTS** (supported for years). Companies stick to LTS.

- Use **JDK 21** (an LTS). That's what this roadmap uses.
- You'll see "JDK", "OpenJDK", "Temurin", "Corretto" — these are all builds of the same Java. Any OpenJDK 21 build is fine. **Eclipse Temurin** is a safe, common pick.

## 6. Setup steps (do these — this is the hands-on part)

### Step 1: Install JDK 21
- Download an OpenJDK 21 build (Eclipse Temurin is easy: search "Adoptium Temurin 21").
- Install it.
- Open a new terminal (PowerShell) and check:

```powershell
java -version
javac -version
```

Both should print a version starting with `21`. If `javac` is missing but `java` works, you installed a JRE, not a JDK — reinstall the JDK.

> Note: If it says "not recognized", the install folder isn't on your PATH. The Temurin installer usually offers a "Set JAVA_HOME" / "Add to PATH" checkbox — tick it. Then open a fresh terminal.

### Step 2: Install IntelliJ IDEA Community
- Download **IntelliJ IDEA Community Edition** (free).
- Install and open it.
- IntelliJ is your main tool, like Visual Studio / Rider for C#. It will detect your JDK. If it asks, point it at your JDK 21 folder.

### Step 3: Run a `main()` from IntelliJ
- New Project → Java → pick JDK 21 → create.
- Make a file `Hello.java` with this:

```java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello from IntelliJ");
    }
}
```

- Click the green ▶ arrow next to `main`. You should see the text print.

**Note on `main`:** `public static void main(String[] args)` is the entry point — where the program starts. C# uses `static void Main(string[] args)`. Almost identical. Java uses lowercase `main` and `String[]` instead of `string[]`.

**Note on the file name:** A public class must live in a file with the **same name**. `public class Hello` must be in `Hello.java`. This is a hard rule in Java (C# doesn't force this).

### Step 4: Run from the terminal (no IDE)
This shows you what IntelliJ does for you behind the scenes.

```powershell
# go to the folder with Hello.java, then:
javac Hello.java      # makes Hello.class
java Hello            # runs it (note: no .class, no .java)
```

You should see `Hello from IntelliJ` (or whatever you printed).

> Tip: `java Hello` takes the **class name**, not the file name. So no `.class` at the end.
