# Session 0.3 — Project Anatomy & First REST App

**Goal:** Learn how a real Java/Spring app is laid out. Understand packages (the word you learned in 0.2). Generate the **CreatorHub** app skeleton with Spring Boot, run it, and write your first REST endpoint. This is where the project starts.

---

## 1. The standard Maven folder layout

Every Maven (and Spring Boot) project uses the same folder shape. You saw part of it in 0.2. Here is the full picture:

```
my-app/
├── pom.xml                      <- build config (deps, build info)
├── src/
│   ├── main/
│   │   ├── java/                <- YOUR CODE goes here
│   │   │   └── com/darshan/creatorhub/...
│   │   └── resources/           <- config + non-code files
│   │       └── application.properties
│   └── test/
│       └── java/                <- YOUR TESTS go here
│           └── com/darshan/creatorhub/...
└── target/                      <- build output (auto-made, ignored by git)
```

Rules to remember:
- **`src/main/java`** — all your real code.
- **`src/main/resources`** — config files, not code. E.g. `application.properties`, SQL files, templates. These get bundled into the jar but are not compiled.
- **`src/test/java`** — tests. Same package layout as `main`, mirrored.

**C# analogy:** C# doesn't force this layout, but it's like keeping your project files under a `src/` and tests under a `Tests/` project. Java just makes it a strict, shared convention so every project looks the same.

## 2. Packages — namespaces for your classes

A **package** is a namespace + a folder path for your classes. It groups related classes and avoids name clashes. This is the word you must keep separate from "dependency".

- **package = C# `namespace`.** Same idea.
- In Java, the package name **must match the folder path**. If a class says `package com.darshan.creatorhub;`, the file must sit in `src/main/java/com/darshan/creatorhub/`.

Example file:

```java
package com.darshan.creatorhub;   // must match folder path

public class App {
    // ...
}
```

### Naming convention
Packages use a **reversed domain name**, all lowercase:
`com.darshan.creatorhub` → company/owner `darshan`, project `creatorhub`.

This is why the groupId in 0.2 looked like `com.darshan`. The groupId usually becomes your base package.

**C# analogy:** `com.darshan.creatorhub` ≈ C# namespace `Darshan.CreatorHub`. Java lowercases it and ties it to folders; C# doesn't force the folder match.

### Importing from other packages
To use a class from another package, you `import` it (like C# `using`):

```java
import com.google.common.base.Joiner;   // like: using Google.Common...
```

Classes in the **same** package don't need an import.

## 3. What is Spring Boot? (short version)

**Spring** is the most popular Java framework for building backend apps. **Spring Boot** is Spring made easy — it sets up sensible defaults so you can run an app fast, with almost no config.

Two things Spring Boot gives you that matter today:

1. **Auto-configuration** — it wires up common things for you (a web server, JSON handling, etc.) based on what dependencies you added. You don't configure a server by hand.
2. **Embedded server** — your app *contains* its own web server (Tomcat). You just run the jar and it serves HTTP. No separate server to install.

**C# analogy:** Spring Boot ≈ **ASP.NET Core**. Both are the "batteries-included" backend framework for their language. The embedded Tomcat ≈ Kestrel (the built-in web server in ASP.NET Core). You run one process and it serves web requests.

We'll go deep on how Spring *actually* works (dependency injection, beans) in Phase 2. Today is just: make it, run it, hit an endpoint.

## 4. Embedded Tomcat — why it's a big deal

Old Java web apps built a `.war` file and you had to install a separate server (Tomcat) and deploy the war into it. Extra steps, extra ops pain.

Spring Boot flips this: the server is **inside** your app. You build one **fat jar** (like the one you made by hand in 0.2!) and run it with `java -jar`. It starts the web server itself and listens on a port (default **8080**).

So:
- One command to run: `java -jar app.jar`
- Easy to put in a Docker container (you know Docker) — just run the jar.
- Same everywhere: local, server, cloud.

**You already did the hard part in 0.2** — building a fat jar by hand with the Shade plugin. Spring Boot does that fat-jar step automatically. That "aha" from yesterday is exactly why Boot is nice.

## 5. Generate the CreatorHub skeleton (Spring Initializr)

You don't write a Spring Boot project from scratch. You generate it from **start.spring.io** (called Spring Initializr). It makes the folders, `pom.xml`, and a main class for you.

### Settings to pick
Go to **https://start.spring.io** and set:

- **Project:** Maven
- **Language:** Java
- **Spring Boot:** the default stable 3.x version (not a SNAPSHOT/M version)
- **Group:** `com.darshan`
- **Artifact:** `creatorhub`
- **Name:** `creatorhub`
- **Package name:** `com.darshan.creatorhub`
- **Packaging:** Jar
- **Java:** 21

### Dependencies to add (click "Add Dependencies")
- **Spring Web** — REST endpoints + embedded Tomcat. (required today)
- **Lombok** — cuts boilerplate (getters/setters). Nice to have.
- **Spring Boot DevTools** — auto-restart on code change. Nice to have.

Click **Generate** → it downloads a zip → unzip it into your project location. Open the folder in IntelliJ. It will read `pom.xml` and set up as a Maven project.

> Put it where you keep the real project (not under `practice/`). This is the actual CreatorHub app you build for months. Suggested: a `creatorhub/` folder. We'll confirm the exact spot in the project step.

## 6. What Initializr gives you

Open the generated project. Key files:

**The main class** (`src/main/java/com/darshan/creatorhub/CreatorhubApplication.java`):

```java
package com.darshan.creatorhub;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class CreatorhubApplication {
    public static void main(String[] args) {
        SpringApplication.run(CreatorhubApplication.class, args);
    }
}
```

- `@SpringBootApplication` — one annotation that turns on Spring Boot's magic (auto-config + scanning your packages for components). More on annotations in Phase 2.
- `SpringApplication.run(...)` — starts the app: boots Spring, starts embedded Tomcat, listens on 8080.
- It's still a normal `public static void main` — same entry point idea from 0.1.

**`src/main/resources/application.properties`** — your config file (empty for now). This is where you'll set the port, DB info, etc. later.

### Run it
Click ▶ on the main class (or Maven panel → `spring-boot:run`). Watch the console — you'll see Spring start and a line like `Tomcat started on port 8080`.

Open a browser at **http://localhost:8080**. You'll get a "Whitelabel Error Page" (a 404). That's EXPECTED — you have no endpoints yet. It means the server is up.

## 7. Your first REST endpoint

A **REST controller** is a class that handles HTTP requests and returns data. Make a new class `HelloController` in the same base package:

```java
package com.darshan.creatorhub;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello from CreatorHub";
    }
}
```

What the annotations mean (short — deep dive in Phase 2):
- **`@RestController`** — tells Spring: this class handles web requests and returns data (not HTML pages). Its return value becomes the HTTP response body.
- **`@GetMapping("/hello")`** — this method runs when someone does `GET /hello`.

**C# analogy:** `@RestController` ≈ `[ApiController]` on a controller. `@GetMapping("/hello")` ≈ `[HttpGet("hello")]`. Almost one-to-one with ASP.NET Core Web API.

Restart the app, then open **http://localhost:8080/hello** → you should see `Hello from CreatorHub`.

That's a working REST API. The server took your HTTP GET, ran the method, and sent the string back.

## 8. Key terms, plain English

| Term | Meaning |
|------|---------|
| **package** | Namespace + folder for your classes. Must match folder path. Like C# `namespace`. NOT a dependency. |
| **`import`** | Bring in a class from another package. Like C# `using`. |
| **Spring** | The big Java backend framework. |
| **Spring Boot** | Spring with easy defaults + embedded server. Like ASP.NET Core. |
| **Spring Initializr** | start.spring.io — generates a Boot project for you. |
| **Embedded Tomcat** | The web server that lives inside your app. Like Kestrel. |
| **Whitelabel Error Page** | Spring's default 404 page. Seeing it = server is up, route not found. |
| **`@RestController`** | Marks a class as a REST request handler. |
| **`@GetMapping`** | Maps a method to an HTTP GET path. |
| **`application.properties`** | Main config file, in `src/main/resources`. |
