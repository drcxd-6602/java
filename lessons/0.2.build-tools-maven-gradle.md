# Session 0.2 — Build Tools: Maven & Gradle

**Goal:** Understand what a build tool does and how to use Maven (and know Gradle). This is your #1 gap coming from C#. Good news: you already know the *idea* from NuGet + MSBuild. This is the same idea with different names.

---

## 1. What is a build tool? Why do you need one?

In Session 0.1 you ran one file by hand: `javac Greeter.java` then `java Greeter`. That's fine for one file.

Real projects have hundreds of files and use outside libraries. Doing that by hand is impossible. A **build tool** does all of it for you:

1. **Manage dependencies** — download the outside libraries your code needs (and the libraries *those* need).
2. **Compile** — run `javac` on all your files, in the right order.
3. **Test** — run your tests.
4. **Package** — bundle everything into one shippable file (a `.jar`).
5. **Run** — start your app.

**C# analogy:** This is exactly what **NuGet + MSBuild + `dotnet` CLI** do together. In Java, one tool (Maven or Gradle) does all of it.

The two big Java build tools are **Maven** and **Gradle**. We use **Maven** as the main one because it's simpler to read and very common. You'll also meet Gradle, so I explain it too.

---

## 2. Dependencies — the core idea

A **dependency** is an outside library your project uses. Example: Google's "Guava" library, or Spring.

You don't download JAR files by hand. You *declare* what you want in a config file, and the build tool fetches it from a big online store called **Maven Central** (like nuget.org).

Once downloaded, the library is cached on your machine so it isn't fetched again. In Maven this cache folder is `~/.m2/repository` (on Windows: `C:\Users\<you>\.m2\repository`). Every project shares this one cache.

**C# analogy:** Maven Central ≈ nuget.org. The `.m2` folder ≈ your local NuGet cache.

### Transitive dependencies

If you add Library A, and A needs Library B, the build tool also downloads B automatically. B is a **transitive dependency** — a dependency of your dependency. You don't list B yourself. Same as NuGet.

---

## 3. Maven basics

### The `pom.xml` file

Every Maven project has a file named **`pom.xml`** at its root. POM = "Project Object Model". It describes your project: its name, its dependencies, how to build it.

**C# analogy:** `pom.xml` ≈ your `.csproj` file. Both are XML. Both list dependencies and project info.

A minimal `pom.xml`:

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.darshan</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>

    <properties>
        <maven.compiler.release>21</maven.compiler.release>
    </properties>

    <dependencies>
        <!-- dependencies go here -->
    </dependencies>
</project>
```

### Coordinates: how a library is named

Every library (including your own project) is identified by three things, called **coordinates**:

- **groupId** — who made it, usually a reversed domain. E.g. `com.google.guava`.
- **artifactId** — the name of the library. E.g. `guava`.
- **version** — which version. E.g. `33.0.0-jre`.

Together they point to exactly one library. To add Guava you write:

```xml
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>33.0.0-jre</version>
</dependency>
```

**C# analogy:** Coordinates ≈ a NuGet package's name + version. Java just splits the name into groupId + artifactId.

## 4. Maven lifecycle & commands

Maven works in **phases** that run in order. When you ask for a phase, Maven runs that phase *and all phases before it*.

The main phases, in order:

```
validate  →  compile  →  test  →  package  →  verify  →  install  →  deploy
```

You mostly use these commands:

| Command             | What it does                                                                               |
| ------------------- | ------------------------------------------------------------------------------------------ |
| `mvn compile`       | Compile your source code (`javac` on everything).                                          |
| `mvn test`          | Compile, then run tests.                                                                   |
| `mvn package`       | Compile, test, then bundle into a `.jar` in the `target/` folder.                          |
| `mvn clean`         | Delete the `target/` folder (old build output).                                            |
| `mvn clean package` | Clean first, then do a fresh package. Very common.                                         |
| `mvn install`       | Package, then copy the jar into your local `.m2` cache so other local projects can use it. |

Because phases run in order, `mvn package` *automatically* compiles and tests first. You don't run each step yourself.

**C# analogy:**

- `mvn compile` ≈ `dotnet build`
- `mvn test` ≈ `dotnet test`
- `mvn package` ≈ `dotnet publish` (roughly — makes the shippable output)
- The `target/` folder ≈ C#'s `bin/` folder (build output).

## 5. Dependency scopes

A **scope** says *when* a dependency is needed. You add scope to a dependency. The common ones:

| Scope               | Meaning                                                                     | C# feel                                    |
| ------------------- | --------------------------------------------------------------------------- | ------------------------------------------ |
| `compile` (default) | Needed everywhere: compiling, testing, running. Most deps.                  | A normal package reference.                |
| `test`              | Only needed for tests, not shipped with the app. E.g. JUnit.                | A test-only package.                       |
| `provided`          | Needed to compile, but the runtime already provides it, so don't bundle it. | Rare; think "the server already has this". |
| `runtime`           | Not needed to compile, only to run. E.g. a database driver.                 | A runtime-only dependency.                 |

Example — JUnit is only for tests:

```xml
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.2</version>
    <scope>test</scope>
</dependency>
```

Default scope is `compile`, so if you write no scope, that's what you get.

## 6. Gradle (know it, don't master it yet)

Gradle is the other big build tool. Same job as Maven (deps, compile, test, package), but:

- Config file is **`build.gradle`** (or `build.gradle.kts`), **not XML**. It's a script — shorter and more flexible.
- Gradle is often **faster** (it caches and skips work it already did).
- Android uses Gradle. Many newer Java projects use it too.

The same Guava dependency in Gradle:

```groovy
dependencies {
    implementation 'com.google.guava:guava:33.0.0-jre'
}
```

Notice: the three coordinates (`group:artifact:version`) are on one line. Much shorter than Maven's XML.

**When teams pick which:**

- **Maven** — simpler, very stable, config is easy to read. Great default. Most enterprise Spring apps.
- **Gradle** — when they want speed and flexible custom builds, or it's Android.

For this roadmap we use **Maven**. If a job uses Gradle, the concepts transfer directly — only the file format differs.

**C# analogy:** Maven's XML `pom.xml` ≈ old-style verbose `.csproj`. Gradle's script ≈ a more flexible build script. Both do the same NuGet+MSBuild job.

## 7. Key terms, plain English

| Term                      | Meaning                                                                                    |
| ------------------------- | ------------------------------------------------------------------------------------------ |
| **Build tool**            | Program that fetches deps, compiles, tests, packages, runs. Maven or Gradle.               |
| **Dependency**            | An outside library your code uses.                                                         |
| **Transitive dependency** | A dependency pulled in automatically because another dependency needs it.                  |
| **Maven Central**         | The big online library store. Like nuget.org.                  |
| **`.m2`**                 | Your local folder that caches downloaded libraries.                                        |
| **`pom.xml`**             | Maven's project config file (XML). Like `.csproj`.                                         |
| **Coordinates**           | groupId + artifactId + version — the unique id of a library.                               |
| **Phase**                 | A step in Maven's build order (compile, test, package...).                                 |
| **Scope**                 | When a dependency is needed (compile / test / runtime / provided).                         |
| **JAR**                   | A single packaged file of your compiled Java app/library. Like a `.dll` / packaged output. |
| **`target/`**             | Maven's output folder. Like C#'s `bin/`.                                                   |

## 10. One-line recap

A build tool (Maven) reads `pom.xml`, downloads dependencies from Maven Central into `.m2`, then compiles → tests → packages your code into a jar in `target/`. It's NuGet + MSBuild + `dotnet` rolled into one, with XML config.
